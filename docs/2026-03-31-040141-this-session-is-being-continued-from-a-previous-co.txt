  2. onAuthStateChange 코드가 outdated (Critical)

  ADR의 "변경 전(현재)" 코드:
  // ADR이 가정하는 현재 코드
  if (!session) {
    useAuthStore.setState({ session: null, organizationId: null, plan: null,
  isCloud: false });
  } else {
    useAuthStore.setState({ session });
  }

  실제 현재 코드 (이번 세션에서 수정함):
  // 방금 커밋한 코드
  supabase.auth.onAuthStateChange((_event, session) => {
    useAuthStore.setState({ session });
  });

  session === null 시 org 상태를 초기화하지 않도록 이미 변경했으므로, ADR의
  Phase 3 "변경 후" 코드도 이 기반 위에서 재설계해야 합니다.

  3. tutomate-q 딥링크 스킴 충돌 (Critical - 누락)

  ADR은 tutomate:// 스킴만 다루지만, tutomate-q 앱도 동일 스킴을 쓰면 
  충돌합니다. 두 앱이 같은 기기에 설치되면 OS가 어느 앱에 deep link를 보낼지
  결정할 수 없습니다.

  필요: tutomate-q://auth/callback 별도 스킴 등록 + Supabase Redirect URLs에 둘
  다 추가.

  4. Anonymous → OAuth 유저 전환 방식 미결정 (Critical)

  ADR은 OAuth 로그인 시 새 Supabase 유저가 생성된다고 가정하고,
  create-trial-org(device_id)로 기존 org를 연결합니다. 하지만:

  - Supabase에는 auth.linkIdentity() API가 있어 anonymous 유저에 OAuth 
  identity를 연결할 수 있습니다
  - 이렇게 하면 user_id가 바뀌지 않아 기존 user_organizations 매핑이 그대로
  유지됨
  - device_id swap이 불필요해짐

  두 가지 접근:

  ┌──────────────┬─────────────────────────────────┬───────────────────────┐
  │              │  Option A: 새 유저 생성 (ADR    │      Option B:        │
  │              │              현재)              │     linkIdentity      │
  ├──────────────┼─────────────────────────────────┼───────────────────────┤
  │ user_id      │ 변경됨                          │ 유지됨                │
  ├──────────────┼─────────────────────────────────┼───────────────────────┤
  │ org 연결     │ device_id로 재연결 필요         │ 자동 유지             │
  ├──────────────┼─────────────────────────────────┼───────────────────────┤
  │ 복잡도       │ 높음 (swap 로직)                │ 낮음                  │
  ├──────────────┼─────────────────────────────────┼───────────────────────┤
  │ Supabase     │ -                               │ 네이티브 지원         │
  │ 지원         │                                 │                       │
  └──────────────┴─────────────────────────────────┴───────────────────────┘

  권장: Option B (linkIdentity) — 더 단순하고 안전합니다.

  5. Naver admin.auth.generateLink 방식 리스크 (Medium)

  ADR Phase 4에서 Naver용으로 admin.auth.generateLink({ type: 'magiclink' }) →
  verify URL로 리다이렉트하는 방식을 제안하는데:

  - generateLink로 생성한 magic link는 일회성이고, 사용 시 이메일 인증 완료로
  처리됨
  - 이건 Naver OAuth를 magic link로 "우회"하는 것이라 Supabase의 의도된 사용법이
   아님
  - Supabase의 Custom OIDC provider 또는 auth.admin.createUser + generateLink
  조합이 더 적절

  6. Supabase OAuth 허용 도메인에 커스텀 스킴 제한 (Medium - 누락)

  tutomate://auth/callback을 Supabase Redirect URLs에 추가해야 하는데,
  Supabase가 커스텀 스킴(tutomate://)을 redirect URL로 허용하는지 확인 필요.
  일부 버전에서는 http(s)://만 허용합니다.

  대안: http://localhost:PORT/auth/callback을 사용하고 Electron에서 로컬 서버를
  잠시 띄우는 방식 (VS Code 등이 사용하는 패턴).

  7. 앱 두 개 (tutomate / tutomate-q) UI 변경 누락 (Medium)

  ADR은 App.tsx 하나만 수정한다고 하지만, 실제로는:
  - apps/tutomate/src/App.tsx
  - apps/tutomate-q/src/App.tsx

  둘 다 수정해야 합니다. SettingsPage도 마찬가지.

  8. OAuth URL의 Supabase 도메인 허용 목록 누락 (Low)

  ADR Phase 2의 authHandler.ts에서 허용 도메인:
  const allowed = ['accounts.google.com', 'kauth.kakao.com', 'nid.naver.com'];

  Supabase auth URL(<project>.supabase.co/auth/v1/authorize)도 추가해야 합니다.
  signInWithOAuth({ skipBrowserRedirect: true })가 반환하는 URL은 Supabase
  도메인입니다.

  9. Phase 순서 비효율 (Low)

  Phase 1(OAuth 모듈)을 먼저 만들지만, deep link(Phase 2) 없이는 테스트 불가.
  Phase 2 → Phase 1 → Phase 3 순서가 더 실용적입니다.

  1. Supabase Manual Linking 설정 (Critical - 누락)

  linkIdentity()를 사용하려면 Supabase 대시보드에서 Manual Linking을 활성화해야
  합니다:
  ▎ Authentication > Settings > GOTRUE_SECURITY_MANUAL_LINKING_ENABLED: true

  ADR Action Items에 이 설정이 빠져 있습니다.

  2. initialize() 동시 호출 방지 (Critical - 누락)

  현재 initialize()는 session 유무 체크는 있지만, loading guard가 없어
  onAuthStateChange(SIGNED_IN) + 정상 mount에서 동시 호출 시 race condition 발생
   가능. create-trial-org가 2번 호출될 수 있습니다.

  // 필요한 가드 (ADR에 미반영)
  initialize: async () => {
    if (get().loading) return; // ← 이 가드가 없음
    set({ loading: true });
    // ...
  }

  3. Naver에서 linkIdentity 불가 (Medium - 미언급)

  Naver는 Supabase 네이티브 프로바이더가 아니라 linkIdentity()를 사용할 수
  없습니다. ADR은 이 점을 Phase 4 참고에서 짧게 언급만 하고, 구체적인 전환
  플로우가 없습니다.

  Naver 로그인 시:
  - Edge Function에서 admin.createUser() → 새 user_id 생성
  - 기존 anonymous 유저와 다른 user_id → org 연결 끊김
  - device_id 기반 재연결 필요
